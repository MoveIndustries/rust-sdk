# Maintainers' Guide — `confidential-assets`

This document covers the invariants a maintainer needs to know before
changing this crate. The user-facing README is for callers; this is for
people editing `src/crypto/**`, `src/internal/**`, the BCS layer, or the
test fixtures.

The crate exists in three layers:

1. **Crypto primitives** (`src/crypto/`) — Twisted ElGamal, σ-protocols,
   Fiat–Shamir transcript, range-proof wrapper, Pollard-kangaroo decryption.
2. **Wire format** (`src/bcs.rs`, the `serialize_sigma_proof` helpers in
   each proof module) — byte layout the Move verifier consumes.
3. **API surface** (`src/api/`, `src/internal/`) — entry-function builders
   and view-function readers; relatively low risk by themselves.

The crypto and wire-format layers are where bugs cost money. The API
layer is where bugs cost user time.

---

## 1. Security-critical surface

A change to any of the following warrants extra review and ideally a
fixture-comparison or e2e run before merge:

| Area | File(s) | Why critical |
| --- | --- | --- |
| Transfer σ-proof gen/verify | `src/crypto/confidential_transfer.rs` | Hides amounts; bug here breaks balance integrity |
| Withdraw σ-proof gen/verify | `src/crypto/confidential_withdraw.rs` | Authorizes fund movement out of confidential balance |
| Key-rotation σ-proof | `src/crypto/confidential_key_rotation.rs` | Owns the only path to change decryption key safely |
| Normalization σ-proof | `src/crypto/confidential_normalization.rs` | Required before rollover; must preserve plaintext |
| Registration proof | `src/crypto/confidential_registration.rs` | Binds account ↔ encryption key |
| Range proofs | `src/crypto/range_proof.rs` | Bulletproofs wrapper; non-negativity guarantee |
| Fiat–Shamir transcript | `src/crypto/fiat_shamir.rs` | Byte ordering must match the Move verifier exactly |
| Shared σ helpers | `src/crypto/sigma_helpers.rs` | Recently refactored (commit `8b0df7c`); shared math across proofs |
| Twisted ElGamal | `src/crypto/twisted_el_gamal.rs` | Encryption primitive every other module sits on |
| Encrypted amount layout | `src/crypto/encrypted_amount.rs` | 4×u64 chunked amount; must match Move struct layout |
| Twisted Ed25519 key | `crates/movement-sdk/src/crypto/twisted_ed25519.rs` | Private key; relies on `Zeroize` + redacted `Debug` (commit `d4fd893`) |
| σ-verifier panic resistance | `src/crypto/confidential_transfer.rs` (and friends) | See the regression tests added in `0f28280` — malformed proof points must not panic |

Anything that touches the **Fiat–Shamir transcript byte stream** is
load-bearing for *all* proofs above. If you reorder transcript inputs,
every proof becomes invalid and there is no on-chain way to roll the
state back.

## 2. Wire-format compatibility (must not drift)

These layouts are observed by the Move verifier and/or by the TypeScript
SDK. Changing any of them is a coordinated multi-repo change.

| Wire item | Defined / asserted in | Observed by |
| --- | --- | --- |
| Entry-function arg encoding | `src/bcs.rs::serialize_vector_u8`, `bcs_serialize_move_vector_u8` | Move verifier |
| Transfer σ-proof bytes | `ConfidentialTransfer::serialize_sigma_proof` | Move verifier + TS fixture (`fixtures/transfer_sigma.fixture.json`) |
| Withdraw σ-proof bytes | `ConfidentialWithdraw::serialize_sigma_proof` | Move verifier |
| Key-rotation σ-proof bytes | `ConfidentialKeyRotation::serialize_sigma_proof` | Move verifier |
| Normalization σ-proof bytes | `ConfidentialNormalization::serialize_sigma_proof` | Move verifier |
| Range-proof bytes | `src/crypto/range_proof.rs` | Move verifier |
| Fiat–Shamir transcript order | `src/crypto/fiat_shamir.rs` | Move verifier + TS (`fiat_shamir_ts_tests.rs`) |
| Encrypted-amount ciphertext | `EncryptedAmount::get_ciphertext_bytes` (`src/crypto/encrypted_amount.rs`) | Move verifier |
| Address inputs to transcript | `(chain_id, sender, contract_addr, token_addr)` order, see each proof's `create_with_balance` / `create` | Move verifier |

If you change any of these, the change is **only safe** if all three
peers (Rust SDK, TS SDK, Move module) are updated and re-published in
lockstep. There is no version negotiation.

## 3. Tests proving TS ↔ Rust ↔ Move compatibility

| Compatibility edge | Tests |
| --- | --- |
| **Rust σ self-consistency** (gen ↔ verify roundtrip per proof type) | `tests/proof_tests.rs` |
| **Rust ↔ TS σ parity** (transfer σ) | `tests/transfer_sigma_fixture.rs` against `tests/fixtures/transfer_sigma.fixture.json`, generated by `fixtures/ts/generate.ts` |
| **Rust ↔ TS Fiat–Shamir parity** | `tests/fiat_shamir_ts_tests.rs`, plus pure-Rust transcript invariants in `tests/fiat_shamir_tests.rs` |
| **BCS layout** | `tests/bcs_tests.rs` |
| **Pollard-kangaroo decryption** | `tests/decryption_tests.rs`, `tests/memoize_tests.rs` |
| **Registration proof** | `tests/registration_tests.rs` |
| **σ-verifier panic resistance** (malformed proof points) | regression tests added in `0f28280`, in the proof modules' `#[cfg(test)]` blocks |
| **README API stability** | `tests/readme_compiles.rs` (compiles every snippet documented in `README.md`) |
| **Rust ↔ Move (only authoritative on-chain check)** | `tests/e2e/**` — `#[ignore]`d; run via `scripts/run-ca-e2e.sh` against the published Move module. CI: `.github/workflows/ca-e2e.yml`. |

The TS fixture (`fixtures/ts/generate.ts`) is the only thing that can
catch a silent divergence from the TS SDK. Regenerate it (and re-run
`tests/transfer_sigma_fixture.rs`) any time you touch transfer σ
serialization, the FS transcript, or the encrypted-amount layout.

The e2e suite is the only thing that catches divergence from the Move
verifier.

## 5. Release checklist (before publishing this crate)

The crate is currently `publish = false` in `Cargo.toml`. Before flipping
that:

1. **Fixture freshness.** Regenerate the TS fixture (`fixtures/ts/generate.ts`)
   from the current TS SDK release; confirm `tests/transfer_sigma_fixture.rs`
   passes.
2. **e2e green.** `.github/workflows/ca-e2e.yml` must be passing on the
   commit being released, against the `APTOS_CORE_COMMIT` pin documented
   in §4.
3. **Lints + non-e2e tests.** `cargo clippy -p confidential-assets
   --all-targets --all-features -- -D warnings` and
   `cargo test -p confidential-assets --tests` clean.
4. **README compile-test.** `tests/readme_compiles.rs` must still compile
   — every public snippet in `README.md` must be valid against the
   current API surface.
5. **Version bump rules** (when `publish = true`):
   - any change in §1 or §2 → MAJOR until a stable 1.x is reached;
     coordinated multi-repo release if §2 layout changes.
   - additive API in `src/api/` only → MINOR.
   - doc / test / refactor without surface change → PATCH.
6. **Changelog entry** that explicitly calls out any §2 wire-format
   change and the matching Move module commit.

## 6. Rollback / compatibility rules if the Move module changes

There is no in-protocol versioning for the σ-proof or range-proof byte
layouts: a verifier change is a hard cut.

- **Verifier-compatible changes** (new entry function, new view function,
  bug fix that still accepts the old proof bytes): publish a new SDK
  version, no coordination needed beyond the §4 pin bump.
- **Verifier-breaking changes** (transcript reordering, new proof field,
  different range-proof generators, encrypted-amount layout change):
  treat as a coordinated release.
  - Land the Move module change first, on a *new* commit on
    `confidential-asset-prod`.
  - Update the §4 pin and the matching Rust serializer/transcript code
    in the same PR.
  - Expect the TS SDK to do the same — file an issue against the TS
    repo before merging here.
  - There is no "old proofs accepted on new verifier" path. SDK builds
    that produce old-format proofs will fail on-chain after the cutover;
    bump the crate's MAJOR version so callers must opt in.
- **Rolling back the Rust SDK** without rolling back the Move module is
  only safe if the previous SDK version emitted bytes the *current*
  Move verifier still accepts. Otherwise, do not roll back — fix
  forward.
- **Rolling back the Move module** is the riskier direction and should
  only be done with the original deploying team; SDK behavior alone
  cannot recover state if the verifier rejects in-flight proofs.

When in doubt: re-run the e2e suite on the proposed Rust+Move pair
before publishing anything.
